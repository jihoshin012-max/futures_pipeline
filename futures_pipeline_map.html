ICM is not a folder — it’s the governing layer

The amber strip represents the ICM (Interpreted Context Methodology) agent navigation layer. Before every stage run the agent reads three files in order: CLAUDE.md (global rules, 60-line cap, five pipeline rules in first 20 lines) → stage CONTEXT.md (this stage’s contract, 80-line cap) → program.md (immediate task, 30-line cap). program.md carries three machine-readable fields the driver loop parses: METRIC:, KEEP RULE:, and BUDGET: — changing the optimization target for a stage requires only editing this file the evening before; no Python changes needed. Line caps enforce the lost-in-the-middle mitigation — operative instructions must appear in the first 5 lines of each file.

The center column is the pipeline’s spine

Stages 01→07 run sequentially. Stages 02, 03, and 04 have autoresearch inner loops 🔁 — the agent edits exactly one designated file (feature_engine.py, hypothesis_config.json, or exit_params.json) while a fixed harness evaluates. The harness is off-limits. Stage 05 writes feedback_to_hypothesis.md which is copied to 03-hypothesis/references/prior_results.md — the Stage 05→03 feedback loop shown by the right-side dashed arrow. Build order for Pass 3: Stage 04 driver first, then Stage 02, then Stage 03. Do not build 02 or 03 until Stage 04 overnight test passes cleanly.

Six-layer failure tracking

Failed experiments are tracked simultaneously at multiple levels: (1) per-experiment results.tsv per stage with verdict: reverted; (2) results_master.tsv (24 cols) where n_prior_tests automatically tightens the Bonferroni p-value gate — failed experiments compound into raising the bar for all future experiments; (3) feature_catalog.md — the only layer with a human-curated “Dead ends — do not retry” section read by the Stage 02 agent; (4) feedback_to_hypothesis.md → prior_results.md loop from Stage 05 back to Stage 03; (5) replication_pass:false hard gate in hypothesis_generator.py (Rule 4 — structural enforcement, not just logged); (6) audit_log.md lineage reconstruction for every promotion and deployment.

The coral cluster is the extension point

Adding a new strategy archetype requires files in shared/archetypes/{name}/: feature_engine.py (agent edits), feature_evaluator.py (fixed harness, standard evaluate() interface), simulation_rules.md (transcribed from actual source — never invented), exit_templates.md. Plus: a simulator conforming to run(bar_df, touch_row, config, bar_offset) → SimResult, and a scoring model registered via scoring_adapter.py. The pipeline supports NQ, ES, and GC natively; instruments.md is the only file that changes for a new symbol. The new archetype intake checklist must be completed before Stage 03 autoresearch can run.
