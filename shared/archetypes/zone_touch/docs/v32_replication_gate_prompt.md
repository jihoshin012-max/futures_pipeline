## Purpose

The Python reference trade log contains 331 P1 trades with the frozen v3.2 configuration — dual-model waterfall (A-Eq M1 + B-ZScore M2), verified against raw OHLC bar data. The C++ autotrader must reproduce these trades. Any discrepancy means the C++ implementation has a bug.

⚠️ **Python is the authority.** The reference log was produced by code that passed ground-truth verification (16 trades walked bar-by-bar against raw price data, zero discrepancies, all 20 parameters matched spec). If C++ and Python disagree, C++ is wrong.

---

## Cost Convention Mismatch — CRITICAL

The Python reference log records **gross PnL** (no cost deduction per trade). The C++ autotrader computes **net PnL** (deducts `3t × TotalContracts` per trade).

**All PnL comparisons must account for this:**

```
C++ pnl_ticks + (3 × contracts) = Python total_pnl_ticks
```

For example, an M1 trade (3ct) with Python `total_pnl_ticks = 300`:
- C++ should report `pnl_ticks = 300 - 9 = 291`
- Verification: `291 + (3 × 3) = 300` ✓

For an M2 trade (2ct) with Python `total_pnl_ticks = 152`:
- C++ should report `pnl_ticks = 152 - 6 = 146`
- Verification: `146 + (3 × 2) = 152` ✓

⚠️ **Apply this adjustment on EVERY PnL comparison in all three phases.** A systematic offset of exactly `3 × contracts` on every trade is the cost convention — not a bug. Any OTHER PnL difference IS a bug.

## Phase 2: Trade Replication (entry + exit matching)

**Purpose:** Verify C++ produces the same trades as Python — same entries, same exits, same PnL.

⚠️ **REMINDER: Python is the authority.** `v32_replication_reference_P1.csv` is the ground truth. The C++ trade log (`ATEAM_CSV_TEST_V32_trades.csv`) is compared against it.

### Exit Type Strings

The C++ test mode outputs the **same exit type strings** as Python — no mapping needed:

| exit_type | Mode | Meaning |
|-----------|------|---------|
| `target_2` | M1 | T1+T2 full target hit |
| `stop` | M1 | Stop hit (full stop or BE stop after T1) |
| `time_cap` | M1 | Time cap reached |
| `TARGET` | M2 | Target hit |
| `STOP` | M2 | Stop hit |
| `TIMECAP` | M2 | Time cap reached |
| `EOD` | either | C++ only — 15:50 forced close (not in Python) |

### Join Method

Join Python reference and C++ trade log on **datetime** (within 1 second tolerance). If datetime formats differ, also try joining on `RotBarIndex` (Python) ↔ `signal_idx` (C++).

After joining, immediately check for orphans:
- **Python orphans:** trades in the reference log with no C++ match → C++ missed a trade
- **C++ orphans:** trades in the C++ log with no Python match → C++ took a trade it shouldn't have

Report orphan counts and list each orphan's datetime, mode, and direction before proceeding to per-trade comparison. If orphans exist ONLY due to blackout/EOD rules, document and continue. Any OTHER orphan is a FAIL.

### Per-Trade Verification

For EVERY matched trade (all 331), compare:

| Field | Python column | C++ column | Tolerance |
|-------|--------------|------------|-----------|
| Mode | `mode` | `mode` (M1/M2) | Exact |
| Direction | `direction` (1/-1) | `direction` (LONG/SHORT) | Exact (map 1↔LONG, -1↔SHORT) |
| Entry price | `entry_price` | `entry_price` | ±0.01 |
| Exit price | `exit_price` | `exit_price` | ±0.01 |
| Exit type | `exit_type` | `exit_type` | Exact (same strings, except C++ may have EOD) |
| Contracts | `contracts` | `contracts` | Exact |
| PnL | `total_pnl_ticks` | `pnl_ticks + (3 × contracts)` | ±1 tick |
| Bars held | `bars_held` | `bars_held` | Exact |
| Zone width | `zone_width_ticks` | `zone_width_ticks` | Exact |

⚠️ **PnL comparison uses the cost adjustment.** C++ `pnl_ticks` is net (cost deducted). Python `total_pnl_ticks` is gross. Add `3 × contracts` to C++ before comparing.

**Pass criteria:** All 331 trades match on ALL fields within tolerance. Zero mismatches on mode, direction, exit type, contracts, or bars held. PnL within ±1 tick after cost adjustment.

⚠️ **Expected divergences (NOT bugs):**
- A trade that entered after 15:30 ET in Python but was blocked by C++ (blackout rule) → C++ will have fewer trades. Document which ones.
- A trade that was open at 15:50 ET in Python but force-closed by C++ (EOD rule) → exit_type and exit_price will differ. Document which ones.
- These are deployment rules the Python simulator does not implement. If the ONLY differences are these cases, the gate PASSES with a note.

## Phase 3: Skip Verification and Aggregate Matching

**Purpose:** Verify C++ skips the same signals for the same reasons, and aggregate stats match.

### Skip Matching

Compare C++ skipped log (filtered to `POSITION_OPEN` only) against `v32_replication_skipped_P1.csv`.

**Column name mapping (Python → C++):**
- `touch_datetime` → `datetime`
- `touch_bar_index` → `RotBarIndex`
- `qualified_mode` → `mode`
- `score_bzscore` → `score_bz`
- `blocking_trade_id` → `blocking_trade_rbi`
- `skip_reason` → `skip_reason`

⚠️ **The C++ skipped log contains ALL skip reasons (BELOW_THRESHOLD, M2_NOT_RTH, M2_SEQ_EXCEEDED, M2_TF_EXCEEDED, POSITION_OPEN). The Python skipped log contains ONLY POSITION_OPEN skips.** Filter C++ to `skip_reason == "POSITION_OPEN"` before comparing.

| Metric | Python (skipped log) | C++ (skipped log, POSITION_OPEN only) | Tolerance |
|--------|---------------------|--------------------------------------|-----------|
| Total POSITION_OPEN skips | count from skipped log | count from C++ skipped log (filtered) | Exact |
| Specific skipped touches | each row by RotBarIndex | matching entries | All must match |

⚠️ **REMINDER: Python is the authority.** If C++ skips a touch that Python traded (or vice versa), the position tracking or waterfall logic has a bug.

### Aggregate Matching

| Metric | Python reference | C++ trade log | Tolerance |
|--------|-----------------|--------------|-----------|
| M1 trade count | 100 | ? | Exact |
| M2 trade count | 231 | ? | Exact |
| Total trades | 331 | ? | Exact (or 331 minus blackout/EOD cases) |
