New strategy data guide

What needs to be updated in the pipeline when introducing data for a new strategy. Follow the tabs in order — registration before migration, migration before validation.

ℹ
How the pipeline resolves data paths No stage hardcodes file paths. Every data path flows through three files in order: strategy_archetypes.md names the source_ids → data_registry.md maps them to file patterns → Stage 01 validation resolves actual paths into data_manifest.json. Downstream stages read from the manifest only.

Bar type pairing rule
⚠
Touches are coupled to a specific bar type ZRA touch files contain a BarIndex field — an absolute index into whichever bar series ZRA was configured on. The simulator resolves this via timestamp matching at runtime. If you load touch data against a different bar type than ZRA was configured on, BarIndex maps to wrong rows and entry prices are silently wrong. Always pair touches with the matching bar type.
zone_touch (current)
zone_csv_v2 → paired with bar_data_volume

If ZRA reconfigured to 1-min bars → register as zone_csv_v2_time + pair with bar_data_time

Register data sources before dropping any files. Stage 01 validation reads the registry to know what to look for.

1
Add row(s) to data_registry.md
_config/data_registry.md — one row per new source_id
⌄

Add one row per new data source to the Registered Sources table. The source_id must be unique and match the schema filename you'll create in Step 2.

ℹ
Bar data must use typed source_ids — not a generic bar_data. Use bar_data_volume, bar_data_time, or bar_data_tick depending on bar type.
| {source_id} | {type} | {description} | P1, P2 | {SYMBOL}_{source_id}_*.txt | {stages} | Examples: | bar_data_volume | price | 250-vol OHLCV bars (zone_touch) | P1, P2 | NQ_BarData_250vol_*.txt | 02-features, 04-backtest | | bar_data_time | price | 1-min time OHLCV bars | P1, P2 | NQ_BarData_1min_*.txt | {future strategies} | | orderflow_data | orderflow | Cumulative delta per bar | P1, P2 | NQ_Delta_*.csv | 02-features | | zone_csv_v2_time| touches | ZRA touches on 1-min bars | P1, P2 | NQ_ZRA_1min_*.csv | 02-features, 04-backtest |
⚠
File pattern must match Sierra Chart's actual export filename Check what Sierra Chart names the export before writing the pattern. The glob pattern must match the real filename — the pipeline doesn't rename files.
2
Create schema file(s)
stages/01-data/references/{source_id}_schema.md
⌄

One schema file per source_id. The filename must exactly match the source_id. Stage 01 validation checks every data file against this column contract — validation fails if any required column is missing.

# {Source Name} Schema last_reviewed: {date} # Required columns for {source_id} data files. # Validation fails if any column is missing. # source_id must match data_registry.md exactly. | Column | Type | Description | |----------|--------|--------------------------------------| | datetime | str | Bar datetime YYYY-MM-DD HH:MM:SS | | open | float | Bar open price | | high | float | Bar high price | | low | float | Bar low price | | close | float | Bar close price | | volume | int | Bar volume | # Bar type note (for bar_data_* sources): # source_id determines bar type: # bar_data_volume — 250-vol bars # bar_data_time — 1-min time bars # bar_data_tick — N-tick bars
ℹ
For new bar types (time, tick), create stub schema files even if you don't have the data yet. The stub signals that the source_id is registered and the schema will be populated when data arrives.
3
Add instrument (if new symbol)
_config/instruments.md — only if new instrument symbol
⌄

NQ, ES, and GC are already registered. Only needed if adding a new instrument symbol (e.g. CL, MNQ, MES). No stage may hardcode instrument constants — they must come from this file (Rule 5).

## {SYMBOL} - tick_size: {value} # Minimum price increment - point_value: {value} # Dollar value per full point - cost_ticks: {value} # Round-trip transaction cost in ticks - session_start: {HH:MM} # RTH session start (ET) - session_end: {HH:MM} # RTH session end (ET) - bar_data_prefix: {SYMBOL} # Prefix for bar data filenames

Re-run Stage 01 validation after every data change. This regenerates data_manifest.json — never edit it manually.

⛔
Never hand-edit data_manifest.json It is auto-generated by Stage 01. Any manual edit will be overwritten on the next Stage 01 run. All configuration belongs in data_registry.md and period_config.md.
1
Re-run Stage 01 validation
Regenerates data_manifest.json and validation_report.md
⌄
cd futures-pipeline python stages/01-data/validate.py # Outputs: # stages/01-data/output/data_manifest.json ← updated # stages/01-data/output/validation_report.md ← review this
2
Review validation_report.md
Human checkpoint — do not proceed until clean
⌄

Check the report for the following before advancing to any downstream stage:

  • ○All registered source_ids have files found for P1 and P2
  • ○Schema validation passed for all sources (no missing columns)
  • ○Bar offset computed successfully (timestamp alignment verified)
  • ○Date coverage is complete — no unexpected gaps
  • ○Row counts are plausible for each period
  • ○regime_labels.csv generated covering P1 and P2
⚠
A failed schema check on one source does not block other sources — check the report per source. Fix schema issues in the data files or update the schema contract, then re-run Stage 01.

Update feature_evaluator.py

⌄

If the new data source will be used in feature computation, update the evaluator harness to load it. The evaluator reads paths from data_manifest.json — never hardcode paths.

Key rules
  • →Rule: No stage hardcodes data paths. All paths come from data_manifest.json (Stage 01 output), which is generated from data_registry.md.
  • →Rule: Never hand-edit data_manifest.json. Re-run Stage 01 to regenerate it.
  • →Rule: Touch data is coupled to a bar type via BarIndex. Pair touches with the bar type ZRA was configured on.
  • →Rule: source_id must be unique and match the schema filename in 01-data/references/.
  • →Rule: All instrument constants (tick size, point value, cost_ticks) must come from _config/instruments.md — never hardcoded (Rule 5).
  • →Rule: Period boundaries come from _config/period_config.md only. Stage 01 writes resolved dates into data_manifest.json.
