## THE RULE
No stage hardcodes file paths. All paths come from data_manifest.json (Stage 01 output),
which is generated from this registry.

## Data Type Taxonomy
| type        | description                            |
|-------------|----------------------------------------|
| touches     | Per-touch or per-signal event data     |
| price       | OHLCV bar data                         |
| label       | Derived classification labels          |
| orderflow   | Intrabar volume/delta data             |
| fundamental | Macro/economic data                    |
| alt         | Anything else                          |

## To Add a New Source
1. Drop files into the correct location:
   - For bar data: `01-data/data/bar_data/<bar_type>/` (e.g., `bar_data/volume/`)
   - For all other sources: `01-data/data/<source_id>/`
2. Add row to this table
3. Re-run Stage 01 validation (regenerates data_manifest.json automatically)
4. Human checkpoint: review validation report

**Bar-type pairing rule:** Touch/signal sources generated by ZRA must document which bar type ZRA was configured on. If ZRA is reconfigured to a different bar type (e.g. 1-min, tick), register a new source_id for that touches file (e.g. zone_csv_v2_time) — do not reuse zone_csv_v2. The BarIndex field in touch rows is an absolute index into the bar series ZRA was running on. Loading touches against a different bar type will silently produce wrong entry bars and prices.
