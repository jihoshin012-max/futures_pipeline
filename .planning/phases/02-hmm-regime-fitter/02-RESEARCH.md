## Summary

Phase 2 produces `hmm_regime_fitter.py` — a script that fits a `GaussianHMM` exclusively on P1
bar data (2025-09-16 to 2025-12-14, 77 trading days), then applies the frozen model forward to all
144 P1+P2 trading days to generate `regime_labels.csv`. The model is serialized to
`shared/scoring_models/hmm_regime_v1.pkl`. All three outputs are consumed read-only by downstream
stages.

The regime definition architecture (from `_config/regime_definitions.md`) labels each trading day
on **three independent dimensions**: trend (2 states), volatility (3 states), and macro (2 states).
The independence of the three dimensions means three separate HMMs (or two HMMs plus a hard-coded
macro calendar) are the natural implementation — not a single joint HMM. The `.pkl` file should
serialize all three model objects together as a dict.

The critical correctness constraint is: **P2 rows are never passed to `fit()`**. The pipeline
rules make this a hard violation. After fitting, `predict_proba()` (forward-backward posteriors,
not Viterbi) is used to generate soft state assignments; `argmax` of the posterior selects the
hard label for each day.

**Primary recommendation:** Fit three independent `GaussianHMM` models on P1 daily features;
apply frozen models to P1+P2 to produce `regime_labels.csv`; serialize all three as a dict to
`hmm_regime_v1.pkl`.

### Pattern 1: P1-Only Fit, Forward Application

**What:** Load all bar data, split by P1 boundary date, fit HMM on P1, apply to all data.

**When to use:** Always — this is the mandatory no-look-ahead pattern required by pipeline rules.

```python
# Source: verified against hmmlearn 0.3.3 API
import pandas as pd
import numpy as np
import pickle
from hmmlearn.hmm import GaussianHMM
from pathlib import Path

P1_END = "2025-12-14"  # from _config/period_config.md

# Load and aggregate to daily
bar_df = load_all_bars()          # concat P1 + P2 bar files
daily = aggregate_to_daily(bar_df)  # date, open, high, low, close, volume

# Feature matrix
features = compute_features(daily)  # shape (144, n_features)

# CRITICAL: fit on P1 rows only
p1_mask = daily['date'] <= pd.Timestamp(P1_END).date()
X_p1 = features[p1_mask]          # shape (77, n_features)
X_all = features                   # shape (144, n_features)

model = GaussianHMM(n_components=N_STATES, covariance_type='diag',
                    n_iter=100, random_state=42)
model.fit(X_p1)                    # P2 NEVER touches fit()

assert model.monitor_.converged, "HMM did not converge"

# Apply frozen model forward
proba = model.predict_proba(X_all)   # forward-backward posteriors, NOT Viterbi
labels = proba.argmax(axis=1)        # hard label per day
```

### Pattern 4: Feature Engineering from Tick Bars

**What:** Aggregate tick bars to daily OHLCV, then compute regime features.

```python
def aggregate_to_daily(bar_df: pd.DataFrame) -> pd.DataFrame:
    """Aggregate tick bars to daily OHLCV. bar_df columns: Date, Time, Open, High, Low, Last, Volume"""
    bar_df['dt'] = pd.to_datetime(
        bar_df['Date'].str.strip() + ' ' + bar_df['Time'].str.strip(), format='mixed'
    )
    bar_df['date'] = bar_df['dt'].dt.date
    daily = bar_df.groupby('date').agg(
        open=('Open', 'first'),
        high=('High', 'max'),
        low=('Low', 'min'),
        close=('Last', 'last'),
        volume=('Volume', 'sum'),
    ).reset_index()
    return daily

def compute_adx(high, low, close, period=14) -> pd.Series:
    """Pure numpy/pandas ADX — no TA library needed (none installed)."""
    # TR, +DM, -DM
    prev_close = close.shift(1)
    tr = pd.concat([high - low,
                    (high - prev_close).abs(),
                    (low - prev_close).abs()], axis=1).max(axis=1)
    plus_dm = (high - high.shift(1)).clip(lower=0).where(
        (high - high.shift(1)) > (low.shift(1) - low), 0)
    minus_dm = (low.shift(1) - low).clip(lower=0).where(
        (low.shift(1) - low) > (high - high.shift(1)), 0)
    atr = tr.ewm(span=period, adjust=False).mean()
    plus_di = 100 * plus_dm.ewm(span=period, adjust=False).mean() / atr
    minus_di = 100 * minus_dm.ewm(span=period, adjust=False).mean() / atr
    dx = 100 * (plus_di - minus_di).abs() / (plus_di + minus_di)
    return dx.ewm(span=period, adjust=False).mean()
```

### Anti-Patterns to Avoid

- **Passing any P2 rows to `fit()`:** Hard violation. Even passing P1+P2 as one sequence to fit() is wrong. Slice on P1_END before fitting.
- **Using `predict()` (Viterbi) for final labels:** Requirements specify filtered probabilities. Use `predict_proba()` and `argmax`. Viterbi is globally optimal but violates the "filtered" requirement.
- **Fitting one joint HMM on all three dimensions:** `regime_labels.csv` requires three independent columns. A joint HMM would produce combined state codes that don't map cleanly to independent dimension labels.
- **Hardcoding data paths:** All paths must be derived from repo root or be documented constants — pipeline rule (no hardcoded paths).
- **Re-running the fitter after P2 data is used:** Once P2 OOS is run, do not re-fit and overwrite `hmm_regime_v1.pkl`. The model is frozen at generation time.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| EM algorithm for HMM | Custom Baum-Welch | hmmlearn GaussianHMM.fit() | Numerical stability, convergence monitoring, handling degenerate states |
| Forward-backward algorithm | Custom implementation | hmmlearn predict_proba() | Verified correct; scaling/log-sum-exp handled internally |
| Model serialization | Custom JSON export | pickle.dump() | Pickle preserves all internal state (means_, covars_, transmat_) exactly |
| ADX computation | TA library | Pure pandas (see pattern above) | No TA libraries installed; ADX formula is simple to implement correctly |

**Key insight:** The EM algorithm has numerical edge cases (degenerate states, singular covariances)
that hmmlearn handles via `min_covar` and regularization. Do not reimplement.

## Common Pitfalls

### Pitfall 1: Data Leakage via Incorrect Date Boundary
**What goes wrong:** Including P2 bars in the fit() call by using `>=` instead of `<=` on the boundary date, or off-by-one on 2025-12-14 vs 2025-12-15.
**Why it happens:** P1 ends 2025-12-14, P2 starts 2025-12-15. An inclusive/exclusive mismatch leaks one day.
**How to avoid:** Use `daily['date'] <= pd.Timestamp('2025-12-14').date()` (inclusive of P1 end). Assert row count equals expected P1 days (77) before calling fit().
**Warning signs:** X_p1.shape[0] != 77 at fit time.

### Pitfall 2: HMM Non-Convergence on 77 Days
**What goes wrong:** With only 77 training observations, EM may not converge within default n_iter=10.
**Why it happens:** Small dataset, multiple restarts needed.
**How to avoid:** Set n_iter=100. Assert `model.monitor_.converged` after fit. If not converged, increase n_iter or reduce n_components.
**Warning signs:** `model.monitor_.converged == False`.

### Pitfall 3: All-One-State Output
**What goes wrong:** HMM collapses all observations to a single state — Stage 01 validation will fail with "all-one-state" check.
**Why it happens:** Degenerate EM initialization, or n_components too high for the dataset.
**How to avoid:** Use `random_state=42` for reproducibility. After predicting, assert `len(np.unique(labels)) == n_components`. If collapsed, reduce n_components or add feature scaling.
**Warning signs:** Stage 01 CONTEXT.md explicitly checks "not all-one-state" in validation checklist.

### Pitfall 4: NumPy ABI Incompatibility with hmmlearn 0.3.3
**What goes wrong:** Import error or segfault due to NumPy 2.x ABI mismatch with hmmlearn's compiled extensions.
**Why it happens:** hmmlearn 0.3.3 is in limited-maintenance; may have been compiled against NumPy 1.x.
**How to avoid:** Add an import smoke test at the top of the script: `from hmmlearn.hmm import GaussianHMM; assert GaussianHMM is not None`. Verified live: import succeeds with numpy 2.3.5.
**Warning signs:** ImportError mentioning `_hmmc` or `numpy` ABI on script start.

### Pitfall 5: Macro Dimension Requires External Calendar
**What goes wrong:** FOMC, CPI, NFP dates are not in the bar data — they must be hard-coded or loaded from a file.
**Why it happens:** Macro events are external facts not derivable from price data alone.
**How to avoid:** Hard-code the known event dates for P1+P2 date range (2025-09-16 to 2026-03-02) as a Python set in the script. This is acceptable for a frozen P1-only model.
**Warning signs:** macro column all-zeros or all-ones (no variation).

### Pitfall 6: pandas 3.0 Compatibility
**What goes wrong:** pandas 3.0.0 dropped some deprecated groupby/agg behaviors from pandas 1.x/2.x.
**Why it happens:** Project has pandas 3.0.0 installed; `groupby().agg()` named-aggregation syntax changed.
**How to avoid:** Use the `named aggregation` syntax shown in Pattern 4 (`agg(open=('Open', 'first'), ...)`). Avoid old-style dict-of-list agg. Verified: mixed format parsing with `format='mixed'` is required for datetime parsing.
**Warning signs:** FutureWarning or TypeError in groupby aggregation.

## Open Questions

1. **State name assignment mapping (trend and volatility)**
   - What we know: `regime_definitions.md` provides semantic thresholds (ADX > 25 = trending; ATR > 1.5x avg = high_vol). HMM discovers latent states numerically.
   - What's unclear: Should state-to-name mapping be hard-coded (by examining model means after fit) or dynamic (argmax of means)?
   - Recommendation: After fit, log `model.means_` for each HMM. Assign names by ordering states on the relevant feature (e.g., sort by mean ADX: highest = "trending"). This is deterministic and logged for audit.

2. **Macro event calendar coverage**
   - What we know: FOMC, CPI, NFP dates must be provided for 2025-09-16 to 2026-03-02.
   - What's unclear: Should these be hard-coded in the script or loaded from a reference file?
   - Recommendation: Hard-code as a Python set in the script. Only 10-15 events in scope; a reference file would be over-engineering. Document the list explicitly in code comments.

3. **Feature scaling before HMM fit**
   - What we know: GaussianHMM learns its own means and covariances — it does not require pre-scaled features. However, very different feature scales can cause slow convergence.
   - What's unclear: Whether ATR (in price ticks, ~200-1200 range) and log return (~0.001-0.05 range) should be standardized before fit.
   - Recommendation: Standardize features (zero mean, unit variance) using P1 statistics only. Apply same scaler to P2 to avoid leakage. Use `(X - X_p1.mean()) / X_p1.std()` inline — no sklearn StandardScaler needed.
