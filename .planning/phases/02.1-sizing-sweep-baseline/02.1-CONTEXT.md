## Phase Boundary

Joint parameter sweep (StepDist x MaxLevels x MaxTotalPosition) on P1a data across all 3 bar types, producing 3 baseline profiles (MAX_PROFIT, SAFEST, MOST_CONSISTENT) as permanent pipeline infrastructure. Includes a new simulator feature: MaxTotalPosition cap on total exposure.

<decisions>
## Implementation Decisions

### Simulator Enhancement: MaxTotalPosition
- MaxTotalPosition is a NEW simulator feature — caps total exposure
- Check in `_check_add()`: `current_position + proposed_addQty > max_total_position` → refuse add entirely (skip, don't reset)
- Position stays unchanged, Level stays unchanged when refused
- `max_total_position=0` means unlimited (no cap) — matches current C++ behavior
- Simulator needs `max_total_position` config parameter

### Sweep Parameters (Locked)
- StepDist: [1.0, 1.5, 2.0, 2.5, 3.0, 3.5, 4.0, 4.5, 5.0, 5.5, 6.0, 7.0, 8.0, 10.0] — 14 values
- MaxLevels: [1, 2, 3, 4, 5] — 5 values (add pattern control)
- MaxTotalPosition: [1, 2, 4, 8, 16, 0] — 6 values (0=unlimited, 1=pure reversal)
- MCS=16 for all runs (isolates ML as sole pattern control)
- InitialQty=1 for all runs
- All 3 bar types
- 14 x 5 x 6 = 420 combinations x 3 bar types = 1,260 raw runs

### Deduplication (Locked)
- MaxTotalPosition=1 produces identical results regardless of MaxLevels (no adds fire) — 70 combos collapse to 14
- MaxTotalPosition=2 with ML>=2 all produce same pattern
- Deduplicate before running; simulator handles naturally but results must be deduplicated before profile analysis

### Profile Identification (Locked — 3 Profiles per Bar Type)
- **MAX_PROFIT** — highest cycle PF regardless of risk
- **SAFEST** — best survival metrics: lowest worst-cycle DD, lowest max-level exposure %, best tail ratio
- **MOST_CONSISTENT** — best risk-adjusted returns: highest Calmar or Sortino ratio, highest winning session %, lowest max DD duration

### NO PF Gating (Locked)
- PF < 1.0 configs are VALID profile candidates
- Flag PF < 1.0 as "requires hypothesis improvement to be viable" but DO NOT exclude
- Hard PF > 1.0 gate deferred to Phase 5 verdict after all hypotheses, TDS, filters applied
- Hypothesis filters (H11, H13, H33, H40, etc.) may transform raw PF 0.7 into filtered PF 2.0

### Reporting Requirements (Locked)
- Top 3 candidates per profile per bar type
- Full metrics: cycle PF, cycle count, worst-cycle DD, max-level exposure %, tail ratio, Calmar ratio, winning session %, max DD duration, StepDist, MaxLevels, MaxTotalPosition
- MaxTotalPosition=1 (pure reversal) results reported separately per bar type — critical finding if reversal outperforms martingale

### Pipeline Framework (Locked — Permanent Infrastructure)
- Profiles stored in `shared/archetypes/rotational/profiles/` (max_profit.json, safest.json, most_consistent.json)
- `rotational_engine.py` accepts `--profile` flag to load baseline config
- `results_master.tsv` rows include `profile` field in extended_metrics
- Profile definitions in `shared/archetypes/rotational/profiles/profile_definitions.json` — new profiles addable without code changes
- All downstream phases run per-profile — 3 parallel tracks

</decisions>

<specifics>
## Specific Ideas

- MaxLevels add patterns: ML=1 flat adds, ML=2 cycle 1,2, ML=3 cycle 1,2,4, ML=4 cycle 1,2,4,8, ML=5 cycle 1,2,4,8,16
- MaxTotalPosition=1 is pure reversal (never adds) — compare against all martingale configs
- MaxTotalPosition=2 with ML>=2 all equivalent — only one add can ever fire before cap
- MCS=16 chosen to isolate ML as sole pattern control (MCS only caps individual addQty, not total position)
- Correction from original sizing sweep: original spec had MCS as the risk dimension, but MCS only caps individual add size, not total position accumulation

</specifics>
