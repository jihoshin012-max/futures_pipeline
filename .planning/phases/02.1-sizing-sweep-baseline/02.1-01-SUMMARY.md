- `level_at_add` for a cap-reset ADD records the post-reset value (0), not the pre-reset current level — this preserves the original code semantics where a cap-reset add records level=0 in the trade row

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Fixed `level_at_add` regression in cap-reset path**
- **Found during:** Task 2 (post-implementation test run)
- **Issue:** Restructuring `_add()` moved `level_at_add = self._level` after the cap-reset branch, causing cap-reset ADDs to record the pre-reset level instead of 0. This broke `TestAddCap::test_add_cap_resets_level_to_zero`.
- **Fix:** Explicitly set `level_at_add = 0` inside the cap-reset branch (before the MTP gate), and `level_at_add = self._level` in the else branch.
- **Files modified:** `shared/archetypes/rotational/rotational_simulator.py`
- **Verification:** All 50 tests pass including the previously-failing TestAddCap test
- **Committed in:** `1cb9456` (Task 2 commit, incorporated into implementation)
