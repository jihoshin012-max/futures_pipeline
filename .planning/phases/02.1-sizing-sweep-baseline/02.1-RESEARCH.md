## Summary

Phase 02.1 extends the existing rotational simulator with a `MaxTotalPosition` cap and runs a joint 3-parameter sweep (StepDist x MaxLevels x MaxTotalPosition) across all 3 bar types. The primary deliverable is 3 permanent baseline profiles (MAX_PROFIT, SAFEST, MOST_CONSISTENT) stored as JSON in a new `profiles/` directory, plus the profile-aware infrastructure extension to `rotational_engine.py`.

The codebase is in excellent shape for this phase. `run_sweep.py` already exists and shows the exact pattern to extend — it loads bar data once per bar type, iterates configs, and writes TSV/JSON. The simulator is well-structured with a clear config contract. The key new work is: (1) add `max_total_position` to the simulator's `_add()` guard, (2) build a new `run_sizing_sweep.py` that iterates 3 parameters, (3) compute extended profile metrics (worst-cycle DD, max-level exposure %, tail ratio, Calmar, Sortino, winning session %), and (4) lay down the `profiles/` infrastructure.

**Primary recommendation:** Extend the existing sweep pattern in `run_sweep.py` — don't rewrite it. Add `MaxTotalPosition` as a one-line guard in `_add()`. Build profile scoring as a standalone function operating on the deduplicated results DataFrame.

### MaxTotalPosition Guard in `_add()`
Current `_add()` computes `add_qty` then unconditionally increments position. The guard must sit BEFORE the position is modified:

```python
def _check_add(self, proposed_qty: int) -> bool:
    """Return True if add is allowed under MaxTotalPosition cap."""
    if self._max_total_position == 0:
        return True  # unlimited
    return (self._position_qty + proposed_qty) <= self._max_total_position

def _add(self, bar_idx, dt, close, logger):
    add_qty = self._initial_qty * (2 ** self._level)
    level_at_add = self._level

    if add_qty > self._max_contract_size:
        add_qty = self._initial_qty
        self._level = 0
        level_at_add = 0
    else:
        self._level += 1

    # MaxTotalPosition cap: refuse add, DO NOT modify position or level
    if not self._check_add(add_qty):
        # Revert level increment that happened above
        if level_at_add != 0 or add_qty != self._initial_qty:
            self._level = level_at_add  # stay at same level
        return  # skip — position unchanged

    self._anchor = close
    self._position_qty += add_qty
    # ... log trade ...
```

**Critical implementation detail:** The level increment in the current `_add()` is done before the cap check. The guard must revert the tentative level change when refusing. Alternatively, structure the check before any level mutation — this is cleaner:

```python
def _add(self, bar_idx, dt, close, logger):
    # 1. Compute proposed qty without mutating state
    proposed_qty = self._initial_qty * (2 ** self._level)
    if proposed_qty > self._max_contract_size:
        proposed_qty = self._initial_qty
        next_level = 0
    else:
        next_level = self._level + 1

    # 2. MaxTotalPosition gate — refuse entirely, no state change
    if self._max_total_position > 0:
        if self._position_qty + proposed_qty > self._max_total_position:
            return  # skip, level stays at self._level

    # 3. Commit state changes
    level_at_add = self._level
    self._level = next_level
    self._anchor = close
    self._position_qty += proposed_qty
    # ... log trade with level_at_add, proposed_qty ...
```

This is cleaner than the "revert" approach and matches the spec precisely: level stays unchanged when refused.

### Deduplication Strategy
Before running, generate the deduplicated combination list:
```python
def get_unique_combos(step_dists, max_levels_list, max_tp_list):
    """Return list of (step_dist, max_levels, max_tp) tuples after dedup."""
    seen = set()
    combos = []
    for sd in step_dists:
        for ml in max_levels_list:
            for mtp in max_tp_list:
                # Normalize: MTP=1 collapses all ML values to ML=1
                effective_ml = 1 if mtp == 1 else ml
                # MTP=2, ML>=2: only one add can fire — all equivalent
                # Keep as-is; simulator handles naturally, dedup post-run results
                key = (sd, effective_ml, mtp)
                if key not in seen:
                    seen.add(key)
                    combos.append((sd, ml, mtp))
    return combos
```
Note: The spec says MTP=1 collapses 70→14 combos. MTP=2 with ML>=2 equivalence is noted but the spec says "simulator handles naturally, results deduplicated before profile analysis" — so the dedup for MTP=2 happens at the results analysis stage, not pre-run. Plan should run all non-MTP=1-duplicated combos and deduplicate results for the profile analysis.

### Estimated Unique Run Count
- MTP=1: 14 unique runs per bar type (ML doesn't matter — collapses to 14 distinct StepDist values)
- MTP=2: 14x1 = 14 unique (all ML>=2 equivalent; plus ML=1 which is identical to MTP=1, already covered)
- MTP in [4, 8, 16, 0]: 14x5 = 70 each = 280 runs
- Total unique per bar type: 14 + 14 + 280 = ~308 (vs 420 raw)
- x3 bar types: ~924 actual simulator calls

At the observed sweep speed (~0.2s per run from the existing sweep), 924 runs x 3 bar types is approximately 10-15 minutes total. No parallelization strictly required but optional.

### Extended Metrics for Profile Scoring
Current `compute_cycle_metrics()` returns: `cycle_pf`, `n_cycles`, `win_rate`, `total_pnl_ticks`, `max_drawdown_ticks`, `avg_cycle_duration_bars`, `sharpe`, `avg_winner_ticks`, `avg_loser_ticks`, `max_consecutive_losing_cycles`.

Profile scoring requires ADDITIONAL metrics not yet in `compute_cycle_metrics()`:

| Metric | Definition | Profile Use |
|--------|-----------|-------------|
| `worst_cycle_dd` | Max drawdown in any single cycle window (not cumulative) | SAFEST |
| `max_level_exposure_pct` | % of cycles that reached max_level_reached == MaxLevels | SAFEST |
| `tail_ratio` | avg_winner / abs(avg_loser) | SAFEST |
| `calmar_ratio` | total_pnl_ticks / max_drawdown_ticks | MOST_CONSISTENT |
| `sortino_ratio` | mean(pnl) / std(pnl[pnl<0]) | MOST_CONSISTENT |
| `winning_session_pct` | % of trading sessions with net positive PnL | MOST_CONSISTENT |
| `max_dd_duration_bars` | Longest drawdown duration in bars | MOST_CONSISTENT |

These must be computed from the `cycles` DataFrame, which contains all needed fields: `net_pnl_ticks`, `max_level_reached`, `max_adverse_excursion_ticks`, `duration_bars`.

**winning_session_pct** requires grouping cycles by trading date — available from `cycles["start_bar"]` mapped through the filtered bars DataFrame, or by tagging cycles with date during simulation. Simplest approach: the cycles DataFrame doesn't currently include a date column. Options:
1. Add date column to cycle records in TradeLogger (access from `bars_df.iloc[start_bar]["datetime"].date()`)
2. Post-hoc: map start_bar through bars DataFrame passed into the extended metrics function

Option 1 is cleaner. Option 2 avoids touching TradeLogger. Since this is a new metric for profile scoring only, option 2 is simpler and keeps changes minimal — pass the filtered bars DataFrame to the extended metrics function.

## Common Pitfalls

### Pitfall 1: Level State Mutation Before MaxTotalPosition Check
**What goes wrong:** Current `_add()` increments `self._level` before checking the cap. If the cap fires after the increment, `self._level` is left at the wrong value.
**Why it happens:** The existing code has no cap at all — level mutation and position mutation are sequential and unconditional.
**How to avoid:** Compute `proposed_qty` and `next_level` as local variables first, check cap, then commit all state changes at once. See the "Commit state changes" pattern above.
**Warning signs:** Test where MTP=2, ML=3, two adds have already fired — third add should be refused and level should remain at 2, not advance to 3.

### Pitfall 2: MTP=1 Not Being Pure Reversal
**What goes wrong:** MTP=1 means initial_qty=1, so position after SEED is 1. Any ADD would make it 2, which exceeds MTP=1. So ALL adds are refused — every cycle is exactly SEED → wait → REVERSAL with no adds. This is correct pure reversal behavior.
**Why it happens:** If the check is `> max_total_position` instead of `>`, a position of exactly 1 with proposed_qty=1 would give 2 > 1 = refused. Correct.
**How to avoid:** Confirm test: MTP=1, ML=5, StepDist=2.0 → zero ADD trades logged, n_adds=0 for all cycles.

### Pitfall 3: MTP=0 Not Being Unlimited
**What goes wrong:** If code does `if self._position_qty + proposed_qty > 0:` (treating 0 as the limit value), no adds will ever fire.
**How to avoid:** Explicit guard: `if self._max_total_position == 0: return True` (unlimited). Test with MTP=0 producing identical results to existing C++ defaults.

### Pitfall 4: Sweep Results Dict Key Collision
**What goes wrong:** Using `(step_dist, max_levels)` as key without `max_total_position` causes collisions when iterating the full 3-param space.
**How to avoid:** Use tuple key `(step_dist, max_levels, max_total_position)` throughout.

### Pitfall 5: "winning_session_pct" Without Date in Cycles DataFrame
**What goes wrong:** Cycles don't currently record a date. Computing session wins requires knowing which calendar date each cycle belongs to.
**How to avoid:** Pass the filtered bars DataFrame to the extended metrics function. Map `cycle["start_bar"]` → `bars_df.iloc[cycle["start_bar"]]["datetime"].date()` to group cycles by date. Note: `bars_df` uses reset-index so `iloc[start_bar]` is correct.

### Pitfall 6: Profile per Bar Type vs Global Profile
**What goes wrong:** The sweep runs all 3 bar types. Each bar type may select a different config as "best." Profiles may be per-bar-type or global — the CONTEXT.md says "3 baseline profiles per bar type" — so each profile JSON should specify which bar type it applies to.
**How to avoid:** Create one profile JSON per bar type per profile name — OR store all bar types in one profile JSON with nested results. The CONTEXT.md says `max_profit.json`, `safest.json`, `most_consistent.json` — singular — so likely one profile = winning config across bar types. Clarify during planning: if bar types disagree strongly, the planner should flag for human decision. Recommend: store best per bar type inside the profile JSON, allow downstream phase to pick.

### Pitfall 7: Worst-Cycle DD vs Cumulative DD
**What goes wrong:** Current `max_drawdown_ticks` in `compute_cycle_metrics()` is the peak-to-trough of the CUMULATIVE PnL series (i.e., overall max drawdown). "worst-cycle DD" for SAFEST profile is the maximum single-cycle loss (worst individual cycle PnL), not cumulative.
**How to avoid:** `worst_cycle_dd = abs(min(pnl))` where `pnl` is `cycles["net_pnl_ticks"].values`. Different metric from the existing `max_drawdown_ticks`. Do not confuse.
