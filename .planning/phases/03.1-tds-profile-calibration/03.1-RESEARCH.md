## Summary

Phase 03.1 calibrates TrendDefenseSystem thresholds against the 3 winning sizing profiles produced by Phase 02.1 (MAX_PROFIT, SAFEST, MOST_CONSISTENT). The TDS was built and integrated in Phase 03 but all parameters remained at spec defaults — they have never been swept. Before Phase 04 can combine the best alpha configuration with the best TDS configuration, TDS must be independently calibrated on each bar type against the known-good profile configs.

The codebase is in an ideal state for this. `run_sizing_sweep.py` is the direct template: it builds a parameter grid, loops over bar types, loads profiles, runs `RotationalSimulator`, computes extended metrics, and writes TSV/JSON output. The same pattern applies here with TDS config dicts substituted for martingale parameter tuples. The `TrendDefenseSystem` class and its integration in `rotational_simulator.py` are complete with 22 tests passing. The `profiles/` directory has the exact parameter sets to use as the base config for each run.

The key calibration insight from the spec (Section 4.5): TDS parameters have wide ranges. Step widen factor spans 1.25–2.0, drawdown budget is expressed in ticks (bar-type-agnostic), velocity is expressed in seconds (already converted to bar counts at init — the critical Phase 03 design decision). This means the same TDS config JSON can be swept across bar types without manual rescaling — the simulator handles the seconds-to-bars conversion at init time using the bar DataFrame's actual timestamp deltas.

**Primary recommendation:** Build `run_tds_sweep.py` modeled exactly on `run_sizing_sweep.py`. For each (profile x bar_type x tds_config) combination: inject profile martingale params + TDS config into the base rotational config, run the simulator, compute survival delta metrics vs the profile's no-TDS baseline, write TSV rows. Select best TDS config per bar_type+profile by ranking on survival improvement with PnL guard (must not degrade total_pnl_ticks by more than X%).

### Pattern 1: Sweep Harness Modeled on run_sizing_sweep.py

**What:** The sweep harness iterates over (profile x bar_type x tds_config_combo), assembles a full config dict from the profile's martingale params plus the current TDS config, runs `RotationalSimulator`, computes metrics, and writes rows.

**When to use:** Mandatory — this is how all parameter searches work in this codebase.

**Key difference from sizing sweep:** The sizing sweep varies martingale params and tests baseline (no TDS). The TDS sweep fixes martingale params from the profile and varies TDS params. This is a pure calibration pass.

**Pattern:**
```python
# Source: run_sizing_sweep.py pattern (verified in codebase)
for profile_name, profile_data in profiles.items():
    for source_id, bar_type_config in profile_data["bar_types"].items():
        base_config = build_base_config(bar_type_config, source_id)
        # Run no-TDS baseline first (or read from profile JSON directly)
        baseline_metrics = profile_data["bar_types"][source_id]

        for tds_combo in tds_combos:
            tds_config = build_tds_config(*tds_combo)
            config = {**base_config, "trend_defense": tds_config}
            sim = RotationalSimulator(config, bars[source_id])
            result = sim.run()
            metrics = compute_tds_extended_metrics(result, baseline_metrics, tds)
            write_row(metrics)
```

### Pattern 4: TDS Survival Delta Metrics

The output TSV must include both raw metrics AND deltas from the profile's no-TDS baseline:

```python
# New function: compute_tds_survival_metrics()
def compute_tds_survival_metrics(
    result: SimulationResult,
    tds: TrendDefenseSystem,
    baseline: dict,    # profile's bar_type metrics (no-TDS)
    bars_df: pd.DataFrame,
    max_levels: int,
) -> dict:
    extended = compute_extended_metrics(result.cycles, cost_ticks, bars_df, max_levels)
    tds_summary = tds.get_summary()

    # Survival deltas (positive = improvement)
    worst_dd_reduction = baseline["worst_cycle_dd"] - extended["worst_cycle_dd"]
    max_level_reduction = baseline["max_level_exposure_pct"] - extended["max_level_exposure_pct"]
    pnl_impact = extended["total_pnl_ticks"] - baseline["total_pnl_ticks"]
    tail_ratio_delta = extended["tail_ratio"] - baseline["tail_ratio"]

    # TDS-specific metrics
    n_td_flatten = len(result.cycles[result.cycles["exit_reason"] == "td_flatten"])
    # l3_recovery_bars_avg: average duration of cycles with td_flatten exit
    td_flatten_cycles = result.cycles[result.cycles["exit_reason"] == "td_flatten"]
    l3_recovery_avg = td_flatten_cycles["duration_bars"].mean() if len(td_flatten_cycles) > 0 else 0.0

    return {
        **extended,
        **tds_summary,  # l1_triggers, l2_triggers, l3_triggers
        "worst_dd_reduction": worst_dd_reduction,
        "max_level_pct_reduction": max_level_reduction,
        "pnl_impact_ticks": pnl_impact,
        "tail_ratio_delta": tail_ratio_delta,
        "n_td_flatten_cycles": int(n_td_flatten),
        "l3_recovery_bars_avg": round(l3_recovery_avg, 1),
    }
```

### Pattern 5: No-TDS Baseline Is Already Computed

The profile JSON files contain the exact metrics computed by `run_sizing_sweep.py`. There is no need to re-run the simulator without TDS — just read the baseline metrics directly from `profiles/{name}.json`. This saves significant compute time.

### Anti-Patterns to Avoid

- **Re-running P2 data:** HOLDOUT_FLAG check is already in `rotational_engine.py`. Never target P2. Always use period="P1a" for calibration.
- **Varying retracement_reset_pct:** The spec's Phase 03-01 design decision locked this: "consecutive_adds resets on qualifying retracement (>= retracement_reset_pct * step_dist in-favor)". Keep at 0.3 to avoid over-fitting this interaction.
- **Sweeping precursor thresholds:** With only `adverse_speed` populating Detector 5, sweeping `speed_threshold` / `regime_accel_threshold` / `band_speed_state` thresholds is meaningless — they won't fire. Sweep `adverse_speed_threshold` only.
- **Building a custom config-builder:** Use the same `rotational_params.json` structure as `run_sizing_sweep.py` does — assemble the full config dict in code and pass directly to `RotationalSimulator(config, bars)`. No file writes needed per run.
- **Measuring PnL in absolute ticks only:** PnL impact varies by bar type (250vol produces ~10x the ticks of 10-sec). Use pnl_impact_ticks as a relative guard (e.g., must not reduce PnL by > 20%) rather than an absolute cutoff.

## Common Pitfalls

### Pitfall 1: TDS Enabled but No TDS in Config
**What goes wrong:** If config has `"trend_defense": {"enabled": false}` or missing `trend_defense` key, `self._tds = None` and no TDS effects occur. Results look identical to baseline.
**Why it happens:** The simulator's `_TDS_AVAILABLE` guard and config `enabled` flag both must be True for TDS to activate.
**How to avoid:** Assert after simulator init that `sim._tds is not None` when TDS sweep is intended. Or verify l1/l2/l3_triggers > 0 for non-trivial configs.
**Warning signs:** TDS run metrics are identical to baseline metrics.

### Pitfall 2: Bar Duration Stats at Wrong Scope
**What goes wrong:** `bar_duration_stats` is computed inside `RotationalSimulator.run()` from the filtered bar DataFrame. If the filter eliminates most bars (e.g., P1a date filter on P1-only data where date range doesn't match), `median_sec` defaults to 10.0, mis-scaling velocity thresholds for vol/tick bars.
**Why it happens:** The bar_duration_stats uses the first 1000 diffs — if the P1a-filtered DataFrame is tiny, the median may be unreliable.
**How to avoid:** Always run on P1a with the correct bar file paths from `rotational_params.json`. Verify `result.bars_processed` is plausible (>1000 bars) before trusting TDS calibration.
**Warning signs:** 250vol or 250tick results show identical l2_triggers to 10sec runs despite very different bar cadence.

### Pitfall 3: Dedup is Not Needed (TDS Sweep Has No Collapsing Equivalences)
**What goes wrong:** The sizing sweep had deduplication because MTP=1 collapses MaxLevels. TDS has no analogous collapsing equivalence — each parameter combination is genuinely distinct.
**Why it happens:** Copy-paste from sizing sweep without thinking through equivalences.
**How to avoid:** Do not add deduplication logic. Every TDS combo is unique.

### Pitfall 4: Precursor Composite Detector Fires Rarely Without H33/H38/H40
**What goes wrong:** With baseline profiles (no active_filters), static features `price_speed`, `regime_transition_speed`, `band_speed_state` are all 0 or absent. Detector 5 can only fire via `adverse_speed` (H36, a dynamic feature). The precursor composite requires `precursor_min_signals >= 2` by default — with only one signal available, it will never fire.
**Why it happens:** Feature wiring: H36/H39 are dynamic (computed per-bar in simulator). H33/H38/H40 are static (require feature_engine and active_filters config).
**How to avoid:** Either (a) accept that precursor_min_signals=2 will never fire Detector 5 in Phase 03.1 — this is correct behavior for baseline profile calibration; or (b) set `precursor_min_signals: 1` in some sweep variants to test Detector 5 via adverse_speed alone.
**Warning signs:** l1/l2 triggers appear but are entirely from Detectors 1-4 never involving Detector 5.

### Pitfall 5: Comparing Across Bar Types on Raw Tick Counts
**What goes wrong:** 250vol produces ~8532 cycles (SAFEST profile) vs 100 cycles for 10sec SAFEST. Raw worst_cycle_dd values are incomparable across bar types.
**Why it happens:** Different bar types have vastly different cycle counts and PnL magnitudes.
**How to avoid:** Report deltas (improvement over baseline) rather than absolute values when comparing across bar types. Selection logic should be per-bar-type.

### Pitfall 6: TDS Level 3 Cooldown Creates Artificial FLAT Periods
**What goes wrong:** With aggressive Level 3 thresholds (small drawdown_budget_ticks), TDS fires frequently and the strategy spends significant time flat (cooldown). This reduces n_cycles and total_pnl, which looks like "TDS is bad" but is actually expected behavior.
**Why it happens:** TDS Level 3 is designed to force flat. Very aggressive thresholds (drawdown_budget=25 ticks) will trigger frequently on normal cycles, not just runaway moves.
**How to avoid:** Track `l3_trigger_count` per run. Flag configurations where l3_trigger_count > 5% of n_cycles as "over-trigger" — these are likely miscalibrated.

### TDS Sweep Parameter Grid

Based on spec Section 4.5 parameter ranges (1.25-2.0 for step widen, 50-500 for cooldown):

```python
# Source: spec Section 4.5 parameter ranges (verified)

# Primary sweep parameters (impact survival most directly)
DRAWDOWN_BUDGET_TICKS = [25, 50, 100, 200, 500]
VELOCITY_THRESHOLD_SEC = [30, 60, 120, 300]
CONSECUTIVE_ADDS_THRESHOLD = [2, 3, 4, 5]
COOLDOWN_SEC = [60, 180, 300, 600]

# Secondary sweep parameters (response tuning)
STEP_WIDEN_FACTORS = [1.25, 1.5, 1.75, 2.0]
MAX_LEVELS_REDUCTION = [0, 1]

# Precursor (limited scope in Phase 03.1 — H33/H38/H40 not active)
ADVERSE_SPEED_THRESHOLD = [0.5, 1.0, 2.0]
PRECURSOR_MIN_SIGNALS = [1, 2]

# Fixed (per spec recommendation and Phase 03 decision)
RETRACEMENT_RESET_PCT = 0.3
```

Full grid: 5 x 4 x 4 x 4 x 4 x 2 x 3 x 2 = 15,360 raw combos per (profile x bar_type).
With 3 profiles x 3 bar_types = 9 combos, total = 138,240 runs.

**This is too large.** The plan should reduce this to a focused 2-phase approach:
1. Phase A: Sweep primary survival parameters only (drawdown_budget_ticks x velocity_threshold_sec x consecutive_adds_threshold x cooldown_sec) with Level 1/2 params fixed at spec defaults. Grid: 5 x 4 x 4 x 4 = 320 combos per (profile x bar_type) = 2,880 total runs. Feasible.
2. Phase B: Take top 10 Phase A configs per bar_type+profile, then sweep Level 1/2 params and precursor. Much smaller grid.

Alternatively, a single-phase sweep fixing step_widen_factor=1.5, max_levels_reduction=1, precursor_min_signals=2 (spec defaults) and sweeping: drawdown_budget x velocity x consecutive_adds x cooldown = 5 x 4 x 4 x 4 = 320 combos per (profile x bar_type). Total: 2,880 runs. This is the recommended approach for Plan 01.

## Sweep Design Decision

### Two-Plan Structure (Recommended)

**Plan 01:** Build harness + run focused primary sweep (320 combos x 9 = 2,880 runs, ~30-90 min on modern hardware). Level 1/2 params at spec defaults. Find best configs per bar_type+profile on survival metrics.

**Plan 02:** Refine top configs from Plan 01 by sweeping Level 1/2 response params (step_widen_factor, max_levels_reduction). This is a much smaller grid. Store best_tds_configs.json as Phase 04 input.

This mirrors the 3-plan structure of Phase 02.1 (harness, sweep, profiles).

## Open Questions

1. **Acceptable PnL degradation threshold**
   - What we know: TDS adds defensive behavior that will reduce gross PnL (by exiting cycles early). Some degradation is expected and acceptable.
   - What's unclear: No project decision exists on the maximum acceptable PnL degradation in exchange for survival improvement.
   - Recommendation: Use a conservative guard of -20% total_pnl_ticks vs baseline. Flag configs with >20% degradation as over-aggressive. Let the planner or user adjust during Plan 02.

2. **Whether tds_results/ or tds_profiles/ should be the output directory**
   - What we know: `tds_results/` was mentioned in Phase 03 RESEARCH.md as the TDS testing output dir and exists with a .gitkeep. `tds_profiles/` is more semantically correct for calibrated profile outputs (mirrors `profiles/` naming).
   - What's unclear: No explicit decision in STATE.md.
   - Recommendation: Create `tds_profiles/` for calibrated configs. Leave `tds_results/` available if future work needs per-level test outputs.

3. **Whether to use `tds_results/` directory from Phase 03 for integration test output**
   - What we know: Phase 03 created `tds_results/.gitkeep` as placeholder.
   - Recommendation: Use `tds_profiles/` for calibration outputs. `tds_results/` can remain for any future debugging/test outputs.
