  <behavior>
    - test_select_best_tds_config: select_best_tds() ranks by (worst_dd_reduction + max_level_pct_reduction) descending, applies PnL guard (pnl_impact_ticks > -20% of baseline total_pnl_ticks), flags over-trigger configs (n_td_flatten_cycles > 5% of n_cycles), returns top config per group
    - test_pnl_guard_excludes_aggressive: configs where pnl_impact_ticks < -20% * baseline_total_pnl are excluded from selection
    - test_over_trigger_flag: configs where n_td_flatten_cycles > 0.05 * n_cycles get over_trigger=True flag
    - test_output_json_schema: best_tds_configs.json has structure {profile: {bar_type: {trend_defense: {...}, survival_deltas: {...}, baseline_comparison: {...}}}}
    - test_no_tds_baseline_row: each (profile x bar_type) includes a "no_tds" baseline entry for comparison
  </behavior>
  <action>
    Add to `run_tds_sweep.py`:

    1. `select_best_tds(sweep_df, profiles)` -> dict. For each (profile x source_id) group:
       a. Filter out over-trigger configs: where n_td_flatten_cycles > 0.05 * n_cycles. Flag but don't silently drop — include in report with over_trigger=True.
       b. Apply PnL guard: exclude configs where pnl_impact_ticks < -0.20 * baseline_total_pnl_ticks (read baseline from profile JSON). The -20% threshold is a conservative guard per RESEARCH.md recommendation.
       c. Rank remaining by composite score: worst_dd_reduction + max_level_pct_reduction (both positive = improvement). Higher is better.
       d. Select top 1 config. Also keep top 3 for the report.
       e. Build output dict with the TDS config params, survival deltas, and baseline comparison.

    2. `generate_calibration_report(best_configs, sweep_df)` -> str. Markdown report with sections:
       - Summary: N configs tested, N excluded by PnL guard, N flagged as over-trigger
       - Per profile x bar_type: best config params, survival deltas, PnL impact, trigger counts
       - Cross-bar-type comparison: does the same TDS config work across bar types, or are they divergent?
       - Top 3 candidates per group (so human can review alternatives)

    3. `write_best_configs(best_configs, output_dir)` -> None. Write `best_tds_configs.json` with structure:
       ```json
       {
         "MAX_PROFIT": {
           "bar_data_250vol_rot": {
             "trend_defense": { "enabled": true, "level_1": {...}, "level_2": {...}, "level_3": {...}, "precursor": {...} },
             "survival_deltas": { "worst_dd_reduction": N, "max_level_pct_reduction": N, "pnl_impact_ticks": N, ... },
             "baseline_comparison": { "baseline_pnl": N, "tds_pnl": N, "pnl_change_pct": N },
             "metadata": { "over_trigger": false, "composite_score": N, "rank": 1 }
           },
           ...
         },
         "no_tds_baselines": {
           "MAX_PROFIT": { "bar_data_250vol_rot": { ... baseline metrics ... }, ... },
           ...
         }
       }
       ```

    4. CLI: Add `--select-best` flag to `run_tds_sweep.py` that reads existing TSV and runs selection + report generation without re-running the sweep (analogous to `--score-profiles` in `run_sizing_sweep.py`).

    Write TDD tests first in `test_tds_sweep.py`. Use synthetic DataFrames for unit tests with known-value rows to verify selection logic, PnL guard, and over-trigger flagging.

    IMPORTANT: Selection is per-bar-type. Do NOT select a single TDS config across all bar types — each bar type may need different thresholds (Pitfall 5 from RESEARCH.md: raw tick counts are incomparable across bar types).
  </action>

  </verify>
