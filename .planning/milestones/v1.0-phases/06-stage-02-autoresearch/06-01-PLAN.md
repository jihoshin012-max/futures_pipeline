  <action>
DECISION REQUIRED: Outcome variable for MWU spread computation.

CONTEXT.md locks "Outcome variable: trade PnL in ticks (same as backtest engine output)." However, computing true per-touch PnL requires running the backtest simulator for each of ~6000 P1 touches per experiment — prohibitively slow for a 300-experiment loop. The touch CSV contains a Reaction column (price movement in ticks post-entry) which is a directional proxy but NOT identical to backtest engine PnL (which accounts for exits, stops, targets).

RESEARCH.md Pitfall 1 flags this as "the most critical open question." The keep threshold (spread > 0.15) was not empirically calibrated against either metric yet.

OPTIONS:

(A) reaction-proxy — Use Reaction column as outcome proxy
  Pros: Fast (no simulator calls). Available in touch CSV. Directionally correct (positive reaction = price moved favorably). Sufficient for feature ranking even if absolute spread values differ from true PnL.
  Cons: Not identical to backtest PnL. Spread threshold (0.15) may need recalibration. Does not account for exit logic, stops, or targets.

(B) simulator-per-touch — Run backtest simulator per touch to get true PnL
  Pros: Exact match to CONTEXT.md specification. Spread values directly comparable to backtest results.
  Cons: Extremely slow (~6000 touches * 300 experiments). Requires reference exit_params config. Adds complexity and a dependency on backtest_engine.py.

(C) precomputed-pnl — Pre-compute PnL column once from a reference backtest, store alongside touch CSV
  Pros: Fast at evaluation time. True PnL values. One-time cost.
  Cons: Requires running a reference backtest first. PnL depends on exit params which may change. Adds a data dependency.

Present these options to the user and await their selection before proceeding to Task 1.

- Add module-level paths: _REPO_ROOT, _P1_TOUCHES_PATH, _P1_BARS_PATH (bar_data/volume/ path for NQ P1 bars)
- Define SAFE_TOUCH_COLUMNS list (entry-time safe columns only — exclude Reaction, Penetration, ReactionPeakBar, ZoneBroken, BreakBarIndex, BarsObserved, and all RxnBar_*/PenBar_* columns)

- **CRITICAL — per-feature entry_time_violation flag (not top-level violation_count):**
  The evaluate_features.py dispatcher (DO NOT MODIFY) only forwards `result["features"]` into feature_evaluation.json (lines 90-93). Any top-level key like `violation_count` is DROPPED and never reaches the driver. Therefore, embed `entry_time_violation: bool` INSIDE each feature result dict. This way the flag flows through the dispatcher's `features_evaluated` list unchanged.

- evaluate() implementation:
  1. Load bar_df_full via load_bars() once
  2. Load touch_df via load_touches()
  3. Split P1a (DateTime <= 2025-10-31 23:59:59) and P1b (DateTime >= 2025-11-01 AND <= 2025-12-14 23:59:59)
  4. Load feature_engine.py via importlib.util.spec_from_file_location from the same directory (Path(__file__).parent / 'feature_engine.py')
  5. If feature_engine.py does not exist: return {"features": [], "n_touches": len(touch_df)}
  6. For each touch in P1a and P1b: truncate bar_df_full.iloc[:int(touch_row['BarIndex'])], filter touch_row to SAFE_TOUCH_COLUMNS only, call compute_features(bar_df_truncated, safe_touch_row). Wrap in try/except for IndexError/KeyError/ValueError — if caught, set violation_detected=True for that feature.
  7. Collect feature values per feature name for P1a and P1b separately
  8. Track violation count per feature (not globally). A feature has entry_time_violation=True if ANY touch row triggered a violation for that feature.
  9. For each feature: compute tercile edges on P1a values, bin P1a and P1b, use the outcome variable selected by user in Task 0 (default: Reaction from the ORIGINAL touch_df) to compute spread = mean(best_bin_outcome) - mean(worst_bin_outcome). The feature values determine the bins; the outcome is from the original touch_df (NOT the safe_touch_row). This is valid because the entry-time guard protects the FEATURE computation; the OUTCOME is the target variable we're predicting.
  10. Determine "best" bin: the bin with highest mean outcome. Compute spread on P1a.
  11. MWU test: compare P1a best-bin outcome distribution vs P1b best-bin outcome distribution (same bin label, cross-period comparison)
  12. Handle degenerate bins (edges[0] >= edges[1]): return spread=0.0, mwu_p=1.0, kept=False
  13. Keep decision: spread > 0.15 AND mwu_p < 0.10 (from feature_rules.md)
  14. Return format:
      ```python
      {"features": [
          {"name": "zone_width", "spread": float, "mwu_p": float,
           "kept": bool, "entry_time_violation": bool}
       ],
       "n_touches": int}
      ```
      NOTE: Do NOT include a top-level "violation_count" key. The per-feature `entry_time_violation` boolean IS the violation signal. It flows through the dispatcher unchanged because it lives inside each feature dict.
