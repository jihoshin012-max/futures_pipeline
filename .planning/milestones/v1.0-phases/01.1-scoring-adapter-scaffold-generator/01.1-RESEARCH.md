## Summary

This phase is a GSD traceability exercise, not a build phase. The implementation (`shared/scoring_models/scaffold_adapter.py`) already exists and has been manually tested. It runs cleanly against the live repository, produces correct output, and handles all four SCAF-27 behaviors: parsing archetypes, appending stubs, creating test files, and writing audit entries.

The research task is to (1) verify the implementation matches SCAF-27's specification exactly, (2) identify any gaps or edge cases not yet covered, and (3) define a formal test that the planner can use as a phase gate. No new code needs to be written unless gaps are found during planning.

The current strategy_archetypes.md has no active archetype entries — only the template placeholder and the Simulator Interface Contract section. The scaffold generator correctly returns "No unregistered adapters found" when run against this state. This means functional verification requires either a controlled test with a temporary fixture or waiting for the first archetype registration (Phase 3+). The planner must choose between these two verification strategies.

**Primary recommendation:** Plan a single task that (1) confirms the implementation matches each SCAF-27 clause, (2) runs a dry-run with a synthetic archetype fixture to verify end-to-end behavior, and (3) writes a formal test that can be re-run as regression coverage. Mark SCAF-27 complete after verification passes.

### Key Implementation Details

**Parser logic (`parse_archetypes`):**
- Splits on `## ` headers, skipping `## [` prefix (template lines)
- Stops at "Simulator Interface Contract" — preserves the parser from reading contract boilerplate as an archetype
- Fields parsed as `key: value` bullet items; key normalized to `lower().replace(" ", "_")`
- `scoring_adapter` field maps to `fields["scoring_adapter"]`

**Unregistered detection (`needs_scaffold`):**
- `REGISTERED_ADAPTERS = {"BinnedScoringAdapter", "SklearnScoringAdapter", "ONNXScoringAdapter"}`
- An archetype with `scoring_adapter: ZoneTouchScoringAdapter` triggers scaffolding
- An archetype with `scoring_adapter: BinnedScoringAdapter` does NOT trigger scaffolding (correct)

**Class name generation (`make_class_name`):**
- `zone_touch` → `ZoneTouchScoringAdapter`
- `zone-touch` → `ZoneTouchScoringAdapter` (handles hyphens)
- Split on `[\s_-]+`, capitalize each part, append `ScoringAdapter`

**Stub generation (`adapter_stub`):**
- Uses `pd.DataFrame` type annotation — valid because scoring_adapter.py already imports `import pandas as pd`; appended stubs share the module-level import
- Stub raises `NotImplementedError` (not `pass`) — matches Phase 01 decision: "Scoring adapter stubs raise NotImplementedError instead of pass/None — prevents silent failure in Phase 4"
- Includes rotation warning for archetype descriptions matching: `rotat|rank|cross-instrument|relative-strength|sector-momentum`

**Test file generation (`create_adapter_test`):**
- Output path: `shared/archetypes/{slug}/adapter_test.py` (slug = archetype name lowercased, spaces/hyphens → underscores)
- For `zone_touch`: writes to `shared/archetypes/zone_touch/adapter_test.py`
- Creates the directory if it doesn't exist (`mkdir(parents=True, exist_ok=True)`)
- Import in test: `sys.path.insert(0, parents[2])` which resolves to `shared/`; then `from scoring_models.scoring_adapter import {cls_name}` — this works via Python 3.3+ implicit namespace packages (verified: import succeeds without `__init__.py`)
- Two test functions: `test_score_returns_series_of_correct_length` (positive, will FAIL until implemented — by design) and `test_score_raises_not_implemented_before_phase4` (passes while stub is unimplemented)

**Audit entry (`append_audit_entry`):**
- Format: `## {timestamp} | MANUAL_NOTE` with subject, detail, human fields
- Matches the `note` command format from `audit_entry.sh`
- Appends to `audit/audit_log.md` — respects append-only invariant (never overwrites)

## Architecture Patterns

### Verification Strategy

Since `strategy_archetypes.md` currently has no active archetype, end-to-end verification requires a synthetic fixture approach:

**Pattern: Temporary fixture test**
```python
# Create a temp copy of strategy_archetypes.md with a test archetype entry
# Run scaffold_adapter.py pointing at the temp file
# Assert: stub class appended to a temp copy of scoring_adapter.py
# Assert: adapter_test.py created at expected path
# Assert: audit entry appended with correct format
# Cleanup: delete temp files
```

This is the recommended verification approach — does not mutate live files, is repeatable, and can be added as a formal regression test.

### Idempotency Pattern

The script is idempotent by design: if the class already exists in scoring_adapter.py, it is skipped with `SKIP: {cls_name} already exists`. This means the script can be run multiple times safely. The verification test should confirm this behavior too.

### File Path Resolution

```
REPO_ROOT = Path(__file__).resolve().parents[2]
# scaffold_adapter.py is at: shared/scoring_models/scaffold_adapter.py
# parents[0] = shared/scoring_models/
# parents[1] = shared/
# parents[2] = REPO_ROOT (C:/Projects/pipeline)
```

All file paths are resolved relative to REPO_ROOT at import time — no cwd dependency.

## Common Pitfalls

### Pitfall 1: Testing Against Live Files
**What goes wrong:** Running the scaffold against the real `strategy_archetypes.md` with a synthetic archetype entry mutates `scoring_adapter.py` and `audit_log.md`, leaving test artifacts in the working tree.
**Why it happens:** The main() function resolves all paths from REPO_ROOT at module load time.
**How to avoid:** Use a test harness that temporarily patches `ARCHETYPES_MD`, `ADAPTER_PY`, and `AUDIT_LOG` to point at temp files, or pass paths as arguments to the functions directly.
**Warning signs:** `audit_log.md` has a spurious entry for a test archetype name; `scoring_adapter.py` has a stub class that was never intentionally added.

### Pitfall 2: Import Path in Generated Tests
**What goes wrong:** Generated `adapter_test.py` uses `sys.path.insert(0, parents[2])` which adds `shared/` to the path, then imports `from scoring_models.scoring_adapter import {cls_name}`. This works in Python 3.3+ implicit namespace packages but will fail if pytest modifies sys.path in unexpected ways.
**Why it happens:** No `__init__.py` in `shared/` or `shared/scoring_models/`.
**How to avoid:** Verify the import works with `python -m pytest` from the repo root before marking as complete. The import was tested in this research session: it succeeds (implicit namespace packages work on Python 3.3+).
**Warning signs:** `ModuleNotFoundError: No module named 'scoring_models'` when running the generated test.

### Pitfall 3: Parser Fragility on Template Lines
**What goes wrong:** The `## [Add first archetype here...]` line starts with `## [` which is correctly skipped by the `not line.startswith("## [")` guard. If a real archetype were registered using `## [zone_touch]` format (with brackets), it would be silently ignored.
**Why it happens:** The template format uses `## [...]` as a visual placeholder cue.
**How to avoid:** The convention is that real archetypes use `## zone_touch` (no brackets). This is consistent with the template format shown in strategy_archetypes.md. No change needed — just document the convention.
**Warning signs:** `parse_archetypes` returns empty list when an archetype exists in the file.

### Pitfall 4: SCAF-27 Completion Without Functional Test
**What goes wrong:** SCAF-27 is marked complete based on the file existing, without a test that demonstrates the four behaviors work correctly.
**Why it happens:** The file was written before GSD traceability was applied.
**How to avoid:** The plan must include a verification task that runs the scaffold against a fixture and asserts all four outputs. This is the purpose of Phase 01.1.

### Direct function test pattern (for verification task)
```python
import sys, tempfile
from pathlib import Path
sys.path.insert(0, 'C:/Projects/pipeline/shared/scoring_models')
import scaffold_adapter as sa

mock_archetypes_md = """# Strategy Archetypes

## zone_touch
- Description: Signal-touch zone strategy for NQ
- Scoring adapter: ZoneTouchScoringAdapter
- Current status: hypothesis stage

## Simulator Interface Contract
"""

with tempfile.TemporaryDirectory() as tmpdir:
    tmp = Path(tmpdir)
    arch_file = tmp / "strategy_archetypes.md"
    adapter_file = tmp / "scoring_adapter.py"
    audit_file = tmp / "audit_log.md"

    arch_file.write_text(mock_archetypes_md)
    adapter_file.write_text("import pandas as pd\n")
    audit_file.write_text("# Audit Log\n")

    # Monkeypatch the module-level constants
    sa.ARCHETYPES_MD = arch_file
    sa.ADAPTER_PY = adapter_file
    sa.AUDIT_LOG = audit_file
    sa.REPO_ROOT = tmp

    result = sa.main()
    assert result == 0
    adapter_text = adapter_file.read_text()
    assert "class ZoneTouchScoringAdapter" in adapter_text
    assert "raise NotImplementedError" in adapter_text
    audit_text = audit_file.read_text()
    assert "MANUAL_NOTE" in audit_text
    assert "zone_touch" in audit_text
```

### Idempotency test pattern
```python
# Run main() twice — second run should skip without duplicating the class
result1 = sa.main()  # generates stub
result2 = sa.main()  # should print SKIP
adapter_text = adapter_file.read_text()
assert adapter_text.count("class ZoneTouchScoringAdapter") == 1
```
