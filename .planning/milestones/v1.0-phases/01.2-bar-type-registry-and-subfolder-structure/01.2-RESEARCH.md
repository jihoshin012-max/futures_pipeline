## Summary

The architecture document (`Futures_Pipeline_Architecture_ICM.md`) specifies the target state explicitly: bar data files must live in typed subfolders under `stages/01-data/data/bar_data/` (e.g., `volume/`, `time/`, `tick/`), and `_config/data_registry.md` must be updated from the current single `bar_data` entry to typed source IDs (`bar_data_volume`, `bar_data_time`, `bar_data_tick`). The current state has two flat files in `bar_data/` (`NQ_BarData_250vol_P1.txt` and `NQ_BarData_250vol_P2.txt`) with no subfolder and a single undifferentiated `bar_data` source_id in the registry.

The mismatch between the architecture's target layout and the current flat layout is the problem this phase solves. The backtest engine config example in the architecture doc already hard-references `bar_data/volume/` (line 415 of the architecture doc), so downstream phases will fail without this structure in place.

Three deliverables are required: (1) physical subfolder creation and file migration, (2) data_registry.md updated with typed source IDs, and (3) a `bar_data_volume_schema.md` reference file in `stages/01-data/references/` whose filename matches the new source_id per the schema-to-source_id naming convention.

**Primary recommendation:** Create `stages/01-data/data/bar_data/volume/`, move both `NQ_BarData_250vol_*.txt` files there, update `data_registry.md` to replace the `bar_data` row with `bar_data_volume`, and create `stages/01-data/references/bar_data_volume_schema.md`. Register `bar_data_time` and `bar_data_tick` as placeholder rows (no files yet) to match the architecture's full registry picture.

## Architecture Patterns

### Naming Convention: source_id = subfolder = schema filename

The pipeline enforces a consistent naming convention across three locations:
1. `data_registry.md` source_id column
2. `stages/01-data/data/<source_id>/` subfolder (for non-bar sources like `zone_csv_v2`)
3. `stages/01-data/references/<source_id>_schema.md` filename

For bar data, the convention is adapted: bar files live in `bar_data/<bar_type>/` rather than `<source_id>/`, but the source_id still encodes the bar type (`bar_data_volume`). The schema file must be named `bar_data_volume_schema.md` to match.

**Evidence:** The architecture doc states: "Rule: one schema file per source_id — filename must match source_id in data_registry.md" (line 732). The existing `zone_csv_v2_schema.md` demonstrates this pattern — its filename matches `zone_csv_v2` in the registry.

### data_manifest.json path format

From `data_manifest_schema.md`:
```json
"path": "stages/01-data/data/{source_folder}/{filename}"
```

After migration, the manifest entries for `bar_data_volume` must reference paths like:
```
stages/01-data/data/bar_data/volume/NQ_BarData_250vol_P1.txt
```

The manifest is generated by Stage 01 validation — it is NEVER hand-edited. Updating the registry is what causes the manifest to update when Stage 01 reruns.

### Placeholder source_id rows for future bar types

The architecture registers `bar_data_time` and `bar_data_tick` even though no files exist for them yet. This is intentional — it allows the registry to document the intended pattern before the data arrives. Placeholder rows should have `periods: none` or be omitted from the periods column rather than claiming P1/P2 coverage they don't have.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead |
|---------|-------------|-------------|
| File migration | Custom Python migration script | `git mv` — preserves history and satisfies git's tracking |
| Manifest update | Hand-edit data_manifest.json | Stage 01 validation regenerates it — only update registry |
| Path validation | Custom path checker | Stage 01 CONTEXT.md validation checklist covers this |

**Key insight:** The entire data path plumbing (manifest → stage consumption) works through `data_registry.md` as single source of truth. Updating the registry is the only required code change; everything else is file system operations.

## Common Pitfalls

### Pitfall 1: Hand-editing data_manifest.json

**What goes wrong:** data_manifest.json becomes stale or inconsistent with registry.
**Why it happens:** It looks like a config file, but it is a generated artifact.
**How to avoid:** Only update `_config/data_registry.md`. Stage 01 validation regenerates the manifest. The manifest does not exist yet (output/.gitkeep only), so this is not currently an issue.
**Warning signs:** If a manifest file exists with the old `bar_data` source_id after the registry change, it was hand-edited.

### Pitfall 2: Forgetting to update bar_data_schema.md filename

**What goes wrong:** Schema-to-source_id naming convention is violated; scaffold_adapter.py's parse patterns and any future validation tooling that uses source_id to find schema files will fail to resolve.
**Why it happens:** The rename is a detail that's easy to miss when focused on file migration.
**How to avoid:** Rename as part of the same plan task that updates the registry.

### Pitfall 3: Leaving the old flat bar_data/ layout referenced anywhere

**What goes wrong:** The backtest engine config (Phase 4) expects `bar_data/volume/` paths. If any reference doc or config example still points to the flat `bar_data/` layout, Phase 4 will have path resolution failures.
**How to avoid:** Search for `NQ_BarData_*.txt` and `bar_data/` references in config examples after migration. The architecture doc already uses the correct `bar_data/volume/` path.

### Pitfall 4: Claiming P1/P2 coverage for placeholder bar types

**What goes wrong:** `bar_data_time` and `bar_data_tick` listed with `P1, P2` in periods column, but no files exist. Stage 01 validation would report FAIL for missing files.
**How to avoid:** Use `none` or `—` for periods on placeholder rows until files are dropped in.

### Pitfall 5: Deleting the .gitkeep in bar_data/ before subfolders exist

**What goes wrong:** Git stops tracking the empty `bar_data/` directory, then the `volume/` subfolder has no parent in git history.
**How to avoid:** Create `volume/`, `time/`, `tick/` before removing any existing `.gitkeep` files. Better: leave all `.gitkeep` files in place.

### test_bar_type_registry.py (Wave 0 test file)

```python
"""Smoke tests for Phase 01.2 — bar type registry and subfolder structure.

Verifies:
- Physical subfolder layout matches architecture spec
- data_registry.md uses typed source_ids (bar_data_volume, not bar_data)
- Schema file naming convention satisfied
"""
from pathlib import Path

REPO_ROOT = Path(__file__).resolve().parents[1]
DATA_ROOT = REPO_ROOT / "stages/01-data/data/bar_data"
REGISTRY = REPO_ROOT / "_config/data_registry.md"
REFERENCES = REPO_ROOT / "stages/01-data/references"


def test_volume_subfolder_exists():
    assert (DATA_ROOT / "volume").is_dir()


def test_time_subfolder_exists():
    assert (DATA_ROOT / "time").is_dir()


def test_tick_subfolder_exists():
    assert (DATA_ROOT / "tick").is_dir()


def test_p1_bar_file_in_volume_subfolder():
    files = list((DATA_ROOT / "volume").glob("*_P1.txt"))
    assert len(files) >= 1, "No P1 bar file found in bar_data/volume/"


def test_p2_bar_file_in_volume_subfolder():
    files = list((DATA_ROOT / "volume").glob("*_P2.txt"))
    assert len(files) >= 1, "No P2 bar file found in bar_data/volume/"


def test_no_bar_files_in_root_bar_data():
    root_txt_files = list(DATA_ROOT.glob("*.txt"))
    assert len(root_txt_files) == 0, f"Bar files found in flat bar_data/: {root_txt_files}"


def test_registry_contains_bar_data_volume():
    text = REGISTRY.read_text(encoding="utf-8")
    assert "bar_data_volume" in text


def test_registry_does_not_contain_bare_bar_data_source_id():
    text = REGISTRY.read_text(encoding="utf-8")
    # Bare "| bar_data |" row must not exist — only typed variants
    import re
    bare_row = re.search(r"\|\s*bar_data\s*\|", text)
    assert bare_row is None, "Bare 'bar_data' source_id row still in data_registry.md"


def test_bar_data_volume_schema_exists():
    assert (REFERENCES / "bar_data_volume_schema.md").exists()


def test_old_bar_data_schema_renamed():
    # bar_data_schema.md should no longer exist (renamed to bar_data_volume_schema.md)
    assert not (REFERENCES / "bar_data_schema.md").exists(), (
        "bar_data_schema.md should be renamed to bar_data_volume_schema.md"
    )
```
